# post-writing

## When to use this skill

Use this skill when the user asks to draft, edit, validate, or publish a blog post for this repository (`ddangchani.github.io`). The site stores posts as MDX files under `content/posts/` and has an optional post-agent workflow for agentic drafts.

## Repository conventions

- Source posts live in `content/posts/<year>/` for regular/manual posts or `content/posts/agentic/<year>/` for posts generated by the post agent.
- Post files use the format `YYYY-MM-DD-slug.mdx`.
- Keep the post body in MDX-compatible Markdown.
- Prefer Korean for explanations unless the user requests another language.
- Use math notation only when needed and set `useMath: true` when the post uses math.
- Local assets referenced from a post must already exist under `/assets/...`; validation fails on missing local assets.
- Avoid unescaped Liquid-style tokens (`{{`, `{%`) outside code fences.

## Required frontmatter

Every post needs YAML frontmatter with these fields:

```yaml
title: Post title
description: Short SEO/summary description
date: "YYYY-MM-DD"
tags:
  - Tag
categories: []
routeSegments:
  - Route Title or Category
teaser: /assets/logos/background.png
coverImage: /assets/logos/background.png
excerpt: Short excerpt
readingTimeMinutes: 5
useMath: false
featured: false
draft: false
canonicalUrl: https://ddangchani.github.io/encoded-route/
```

Notes:

- `title`, `description`, `excerpt`, `routeSegments`, and `canonicalUrl` must be non-empty.
- `tags` and `categories` are arrays. `categories` may be empty.
- `teaser` and `coverImage` may be a valid `/assets/...` path or `null`.
- `canonicalUrl` should correspond to the encoded route derived from `routeSegments`.

## Preferred workflow

1. Inspect nearby posts in `content/posts/<year>/` to match tone, style, and frontmatter conventions.
2. If the user requests a generated draft, prefer the built-in post agent:
   - Preview path and route: `npm run post:agent -- plan --topic "..."`
   - Write a draft: `npm run post:agent -- draft --topic "..."`
   - Publish workflow: `npm run post:agent -- publish --topic "..." --target-branch main`
3. For manual posts, create or edit an MDX file directly under the correct `content/posts/<year>/` directory.
4. If the topic involves recent facts, APIs, laws, product specs, or source-backed claims, browse authoritative sources and include concise references in the post.
5. Keep claims conservative when sources are sparse.
6. Include a final `## References` section when the post uses external sources.

## Validation checklist

Run these checks after creating or editing a post:

```bash
npm run generate:content
npm run validate:content
```

For a broader publishing check, run:

```bash
npm run validate:publish
```

If frontend validation is too expensive or environment-limited, at minimum run the content checks above and explain the limitation clearly.

## Writing checklist

- Start with a short setup or motivation section.
- Use clear headings and a logical progression.
- Add concrete examples, equations, code, or diagrams only when they improve understanding.
- Ensure every referenced local image exists.
- Keep summaries and descriptions concise enough for metadata.
- Recalculate `readingTimeMinutes` when the body length changes substantially.
- Leave `draft: false` only when the post is intended to publish.
